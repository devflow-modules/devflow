This document defines the lifecycle, trust boundaries, consumer model, and intentional limits of the version 1 provider-derived enrichment proposal export (`devflow.provider-derived-enrichment-proposal`, version `1`).

## 1. Purpose

The provider-derived enrichment proposal export is a local, human-auditable snapshot of an enrichment proposal explicitly reviewed and generated by the user.

Its purpose is to support inspection, documentation, portfolio demonstrations and manual audit. It is not an executable command, an import payload or an instruction to mutate CareerBundle or application data.

The file exists because users need a durable, offline record of what ApplyFlow proposed after explicit review ŌĆö without implying that ApplyFlow will ever read the file back, apply it automatically, or treat it as a sync or mutation command.

---

## 2. Lifecycle

The export lifecycle is linear and ends at download. There is no automatic post-download stage.

```txt
provider metadata (Gmail/Calendar via Nango, server-only)
  ŌåÆ derived signals (client-safe, privacy-redacted)
  ŌåÆ explicit runtime preview (opt-in HTTP + UI)
  ŌåÆ manual review (in-memory selection/dismiss)
  ŌåÆ selected signals (review-ready state)
  ŌåÆ enrichment proposal (in-memory builder)
  ŌåÆ export validation (envelope + enrichment contract)
  ŌåÆ browser-side JSON download (Blob + anchor)
  ŌåÆ lifecycle ends
```

| Stage | Owner | Persisted? |
|-------|-------|------------|
| Provider metadata fetch | ApplyFlow server runtime | No raw retention |
| Derived signals | Server ŌåÆ client preview | Ephemeral |
| Manual review | ApplyFlow client UI | In-memory only |
| Enrichment proposal | ApplyFlow client builder | In-memory only |
| Export document | ApplyFlow export builder | Serialized at click only |
| Downloaded file | User filesystem | User-controlled |

After the user saves the file locally, ApplyFlow does not track, reopen, re-import, or process the artifact. The lifecycle is complete.

---

## 3. Trust boundaries

| Boundary | What crosses it | What must not cross |
|----------|-------------------|---------------------|
| **Google/Nango provider boundary** | Ephemeral metadata for derivation | Tokens, refresh secrets, raw bodies |
| **Server runtime boundary** | Client-safe derived signals | Provider IDs, connection IDs, raw payloads |
| **Client-safe derived signal boundary** | Reviewable signal summaries | Gmail subjects/snippets, calendar descriptions |
| **Manual review boundary** | User selection/dismiss decisions | Automatic apply, persistence |
| **Enrichment proposal boundary** | Canonical `CareerBundleUnifiedSyncEnrichment` proposal | CareerBundle mutation, application updates |
| **Export serialization boundary** | Allowlisted v1 JSON document | UI-only fields, forbidden keys |
| **Local filesystem boundary** | User-owned JSON file | ApplyFlow storage, server upload |

After download:

- The file leaves ApplyFlow control.
- The user controls storage and sharing.
- ApplyFlow does not track the file.
- ApplyFlow does not reopen it.
- ApplyFlow does not import it.

---

## 4. Producer

Only the internal ApplyFlow export builder (`buildProviderDerivedEnrichmentProposalExport`) officially produces version 1 documents in the product.

The standalone validator (`validateProviderDerivedEnrichmentProposalExportV1` in `@devflow/career-sync`):

- Validates structure against the v1 envelope and enrichment contract.
- Does **not** prove authorship.
- Does **not** prove the file was created by ApplyFlow.
- Does **not** verify a signature.
- Does **not** verify cryptographic integrity.

A structurally valid document may have been hand-edited, synthesized, or produced outside ApplyFlow. Validation confirms shape and safety flags ŌĆö not provenance or trust for mutation.

## 6. Validation semantics

**Validated does not mean:**

- Sanitized ŌĆö the validator does not strip or normalize fields.
- Authenticated ŌĆö no signature or provenance check.
- Safe to apply ŌĆö structural validity is not a mutation authorization.
- Importable ŌĆö export-only is an explicit product decision.
- Forward-compatible ŌĆö only version `1` is accepted; future versions are rejected until explicitly supported.

**Validated means** the document:

- Respects the version 1 envelope (strict root allowlist).
- Has `schema` = `devflow.provider-derived-enrichment-proposal` and `version` = `1`.
- Has canonical UTC ISO timestamps for `exportedAt` and `generatedAt`.
- Has fixed safety flags (`reviewRequired === true`; persistence/apply flags `=== false`).
- Contains no prohibited keys (recursive forbidden-key scan).
- Contains enrichment that passes `validateCareerBundleUnifiedSyncEnrichment` with `rejectProviderIdentifiers: true`.

See [PROVIDER-DERIVED-ENRICHMENT-PROPOSAL-EXPORT-VALIDATION.md](./PROVIDER-DERIVED-ENRICHMENT-PROPOSAL-EXPORT-VALIDATION.md).

## 7. Data minimization

### Fields retained in the export envelope

| Field | Role |
|-------|------|
| `schema` | Document type literal |
| `version` | Contract version (`1`) |
| `exportedAt` | UTC timestamp at explicit download click |
| `generatedAt` | UTC timestamp copied from in-memory proposal |
| `sourceSignalCount` | Count of signals that contributed |
| `reviewRequired` | Always `true` |
| `persistedByApplyFlow` | Always `false` |
| `appliedToCareerBundle` | Always `false` |
| `appliedToApplications` | Always `false` |
| `enrichment` | Allowlisted `CareerBundleUnifiedSyncEnrichment` |

### Fields deliberately removed or never included

| Excluded | Reason |
|----------|--------|
| `selectedSignalIds` | Review UI state; count is sufficient for export |
| `sourcePreviewFingerprint` | Staleness control for review UI only |
| Provider IDs | Privacy / cross-boundary safety |
| Connection IDs | Credential-adjacent |
| Tokens | Never in client or export surfaces |
| Provider payloads | Raw data boundary |
| Raw Gmail metadata | Subject, snippet, body |
| Raw Calendar metadata | Description, location, meeting links |
| Attendee emails | PII |
| Organizer emails | PII |

Enrichment is rebuilt field-by-field from the in-memory proposal; unrestricted spreads and `JSON.stringify(proposal)` are not used.

---

## 8. Safety invariants

Every official v1 export produced by ApplyFlow satisfies:

| Invariant | Value / rule |
|-----------|----------------|
| `reviewRequired` | `=== true` |
| `persistedByApplyFlow` | `=== false` |
| `appliedToCareerBundle` | `=== false` |
| `appliedToApplications` | `=== false` |
| `sourceSignalCount` | integer `>= 1` |
| `schema` | literal `devflow.provider-derived-enrichment-proposal` |
| `version` | literal `1` (number) |
| Timestamps | Canonical UTC: `new Date(v).toISOString() === v` |
| Enrichment | Validated in strict mode with `rejectProviderIdentifiers: true` |

Documents that fail any invariant must not be serialized for download in ApplyFlow.

---

## 9. Non-goals

The version 1 export is explicitly **not**:

- A backup of ApplyFlow or CareerBundle state.
- A complete CareerBundle export.
- A guaranteed interoperability format between products.
- An import format.
- A public write API.
- A sync event or webhook payload.
- An executable instruction to ApplyFlow or providers.
- Cryptographic proof of origin or integrity.

---

## 10. Human audit checklist

Use this checklist when manually reviewing a downloaded file:

- [ ] `schema` is `devflow.provider-derived-enrichment-proposal`
- [ ] `version` is `1`
- [ ] `exportedAt` and `generatedAt` are valid canonical UTC ISO strings
- [ ] `sourceSignalCount` is a positive integer
- [ ] `reviewRequired` is `true`
- [ ] `persistedByApplyFlow` is `false`
- [ ] `appliedToCareerBundle` is `false`
- [ ] `appliedToApplications` is `false`
- [ ] No raw provider data (subjects, snippets, bodies, descriptions, locations, meeting links)
- [ ] No tokens or connection IDs
- [ ] Content reflects a derived enrichment **proposal**, not a full CareerBundle

Optional: run `validateProviderDerivedEnrichmentProposalExportV1` programmatically for structural confirmation.

---

## 11. Storage and sharing guidance

- The user controls where the file is stored after download.
- The file may contain career-related derived information (company hints, process stages, signal summaries).
- Do not share publicly without reviewing contents first.
- Do not send through insecure channels if the content is sensitive.
- Do not treat the file as a credential ŌĆö it contains no OAuth tokens but may still be personally identifiable.
- Do not assume ApplyFlow keeps a copy ŌĆö only the user's local copy exists unless they back it up elsewhere.

---

## 12. Historical files

Older exports may contain `enrichment.combinedSignals[].id` values with legacy prefixes such as `gmail-sandbox-*` or `calendar-sandbox-*`.

Those files:

- Remain structurally valid if they pass the v1 validator.
- Do not contain provider IDs because of the prefix ŌĆö the prefix was a historical naming convention, not proof of sandbox provider origin.
- Must not be interpreted as evidence that live provider data was used.

Current exports use runtime-neutral IDs per [PROVIDER-DERIVED-SIGNAL-ID-CONTRACT.md](./PROVIDER-DERIVED-SIGNAL-ID-CONTRACT.md).

---

## 13. Versioning policy

- Incompatible document changes require a new `version` number.
- Version `1` remains strict: extra top-level fields are invalid (`unexpected_root_field:<key>`).
- Future versions are not accepted silently ŌĆö unknown versions return `unsupported_version`.
- There is no promise of future import support when a new version ships.
- New optional top-level fields in a future version require explicit contract review and ADR update.

---

## 14. Import decision

Version 1 is intentionally export-only.

The current product roadmap does not include reading this file back into ApplyFlow, applying it to CareerBundle, updating applications or triggering provider workflows.

Any future import capability would require a separate threat model, product requirement, parser contract, migration policy, user confirmation flow, idempotency rules, provenance model and mutation boundary.

---

## 15. Reconsideration criteria

Import may return to the roadmap only when all of the following are addressed in dedicated design work:

| Criterion | Why it matters |
|-----------|----------------|
| Proven real use case | Avoid speculative complexity |
| Clear user or business benefit | Justify mutation surface expansion |
| Specific threat model | File tampering, prototype pollution, oversized payloads |
| Isolated parser | No ad-hoc `JSON.parse` in UI or routes |
| Size/depth limits | DoS and memory safety |
| Prototype pollution protection | Safe object traversal |
| Version policy | Explicit dispatch per version |
| Migration between versions | Upgrade path for stored files |
| Provenance model | Who created the file, when, and whether it was tampered |
| Idempotency rules | Re-import must not duplicate effects |
| Change preview | User sees diff before apply |
| Explicit confirmation | No silent mutation |
| Rollback | Recovery from mistaken apply |
| Audit trail | Who applied what and when |
| No automatic application | Human gate preserved |

None of the above is implemented in the current cycle.

---

## 16. Current decision

Current decision: keep the version 1 export as a local, human-auditable artifact.

No parser, importer, uploader, persistence layer or application workflow is planned in the current implementation cycle.

See [ADR-002: Keep provider-derived enrichment proposal v1 export-only](../../adr/ADR-002-ENRICHMENT-PROPOSAL-EXPORT-ONLY.md).
